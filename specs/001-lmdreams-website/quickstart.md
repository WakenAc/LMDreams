# Quickstart - LMDreams Institutional Website

## Purpose

This guide describes the planned developer and content workflow for the static Astro website. It is intentionally reproducible and contains no real credentials.

## Prerequisites

- Node.js `24.x` LTS
- npm
- Git

## Install

```bash
npm ci
```

## Run Locally

```bash
npm run dev
```

## Verify Types and Astro Constraints

```bash
npm run lint
npm run check
```

## Run Tests

```bash
npm run test:unit
npm run test:e2e
npm run test:a11y
npm run test:links
npm run test:lighthouse
```

## Build for Production

```bash
npm run build
```

## Preview Production Output Locally

```bash
npm run preview
```

## How to Add a Service

1. Create a new Markdown entry in `src/content/services/`
2. Fill the required schema fields:
   - `title`
   - `slug`
   - `summary`
   - `description`
   - `benefits`
   - `displayOrder`
   - `published`
3. Add a local service image only if it is approved and honest
4. Run:

```bash
npm run lint
npm run check
npm run build
```

If a published service misses required fields, the build must fail.

## How to Add a Project

1. Create a new Markdown entry in `src/content/projects/`
2. Fill the required project fields:
   - `title`
   - `slug`
   - `summary`
   - `locationGeneral`
   - `interventionType`
   - `services`
   - `challenge`
   - `workPerformed`
   - `result`
   - `images`
   - `completionDate`
   - `published`
   - `authorizedForPublication`
3. Reference only approved local images
4. Run:

```bash
npm run lint
npm run check
npm run build
```

## How the current implementation treats pending content

- Services are currently represented by clearly flagged `[EXEMPLO]` entries
- Projects stay unpublished until they have at least 2 real authorized photos
- Direct phone/email links stay absent until confirmed by the company
- The contact form UI exists now, but only becomes live after configuring a real Formspree endpoint

These are intentional truthfulness safeguards, not missing features.

## How to Approve a Project for Publication

A project may only go public when:

- `published = true`
- `authorizedForPublication = true`
- required fields are complete
- its images are visually truthful and authorized

Changing only one of these flags is not enough if the content or visuals are incomplete.

## How to Add Images

### Preferred placement

- Keep optimized, publishable images under `src/assets/` so Astro can optimize them
- Keep only unprocessed static files such as `favicon.svg` under `public/`

### Required image practices

- Define informative `alt` text where relevant
- Reserve dimensions to avoid layout shift
- Prefer modern formats such as AVIF/WebP when generated by Astro
- Avoid serving oversized originals directly

## How to Identify Image Origin

Every project/service image must map to one of these origins:

- `real-project`
- `company-photo`
- `stock`
- `generated`
- `illustrative`
- `placeholder`

`generated`, `stock`, `illustrative`, and `placeholder` assets must never be presented as proof of a real completed LMDreams project.

## How to Configure the Contact Form

1. Create the form in Formspree
2. Obtain the public endpoint in the format:

```text
https://formspree.io/f/YOUR_FORM_ID
```

3. Store that endpoint in the centralized company/site configuration
4. Verify:
   - success path
   - failure path
   - honeypot field
   - fallback `tel:` and `mailto:` links

No secret key is required in the frontend for the selected v1 approach.

If the endpoint is absent, the implemented UI renders an explicit unavailable state instead of a broken live form.

## How to Configure GitHub Pages

### Project Pages mode

Use this when the site will live at:

```text
https://<user>.github.io/<repo>/
```

Set:

- `siteUrl` to the full GitHub Pages URL
- `basePath` to `/<repo>`
- no `public/CNAME`

### Custom domain mode

Use this when the site will live at:

```text
https://dominio-da-empresa.pt/
```

Set:

- `siteUrl` to the real confirmed domain
- `basePath` to `/`
- add `public/CNAME` with the confirmed domain only

DNS remains an external operational step.

## How to Execute Deployment

1. Merge validated changes to the default branch
2. Ensure repository Pages source is set to GitHub Actions
3. Let `deploy-pages.yml` run
4. Confirm the exposed deployment URL
5. Smoke test:
   - homepage
   - `/empresa/`
   - `/empresa/como-trabalhamos/`
   - `/servicos/`
   - `/projetos/`
   - `/contactos/`
   - `/privacidade/`
   - a direct detail page (if any)
   - `404.html`

## How to Verify Publication

Check:

- no missing images
- no broken internal links
- metadata present
- `robots.txt` and sitemap available
- keyboard navigation works
- contact alternatives visible
- example service content replaced or intentionally kept non-public
- Formspree endpoint configured if the public form is meant to be active
- no placeholder content exposed as real facts

## How to Replace a Higgsfield-Generated Asset with a Real Photo

1. Add the approved real image to the correct local asset folder
2. Update the content entry to point to the real image
3. Change the origin metadata from `generated`/`illustrative` to the appropriate truthful value
4. Update provenance notes in the planned design provenance document
5. Re-run checks and build

## How to Avoid Publishing Placeholders

- Never commit fake contact data as if it were real
- Keep unconfirmed blocks unpublished or explicitly pending
- Do not publish project entries without authorization
- Replace illustrative service/project visuals before using them as proof-like portfolio media

## How to Diagnose Asset Path Problems

1. Confirm `siteUrl` and `basePath` match the selected Pages mode
2. Run a production build locally
3. Preview the built site
4. Check broken assets on:
   - homepage
   - a direct detail route
   - 404 page
5. Re-check canonical URLs, OG images, and sitemap output when the public URL changes
