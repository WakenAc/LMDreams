# Phase 0 Research - LMDreams Institutional Website

## Research Scope

This document resolves the material technical decisions required for the implementation plan while preserving full compliance with the LMDreams constitution.

## D01 - Astro Version

**Decision**: Use Astro `7.0.8`, the current stable release observed from the npm registry.

**Rationale**:
- Matches the user preference for Astro stable
- Supports static output, file-based routing, content collections, and `astro:assets`
- Works with modern TypeScript and official integrations such as `@astrojs/sitemap`

**Alternatives considered**:
- Older Astro 6.x - rejected because there is no project constraint requiring an older version
- Another static site generator - rejected because the preferred stack explicitly centers Astro

**Risks**:
- Newer major versions can introduce subtle config differences

**Mitigation**:
- Pin the exact version in `package-lock.json`
- Keep the implementation limited to core, well-documented Astro features

**Constitutional impact**: Positive. Astro enables static GitHub Pages deployment without introducing backend runtime.

## D02 - Node.js and Package Manager

**Decision**: Use Node.js `24.x` LTS and npm with a committed `package-lock.json`.

**Rationale**:
- Astro `7.0.8` declares `node >=22.12.0`
- Node `24.x` is LTS and supported by Playwright and Vitest
- npm aligns with the requested package manager and lockfile strategy

**Alternatives considered**:
- Node `22.x` LTS - valid fallback, but Node 24 is the newer LTS and lines up with current Astro deployment guidance
- pnpm or yarn - rejected because npm is the preferred package manager

**Risks**:
- Contributors may have older local Node installations

**Mitigation**:
- Document the exact Node major in `quickstart.md` and CI setup
- Add an engines field during implementation

**Constitutional impact**: Positive. Reproducible installs support maintainability and pre-publication validation.

## D03 - Rendering Architecture

**Decision**: Implement the website as a static multi-page Astro application using file-based routing and build-time HTML generation for every public page.

**Rationale**:
- Fully compatible with GitHub Pages
- Delivers indexable HTML without JavaScript dependency
- Fits the institutional-content nature of the website
- Simplifies accessibility, SEO, and failure isolation

**Alternatives considered**:
- SPA shell with client-side routing - rejected because it weakens static-first SEO/accessibility and violates the explicit non-SPA requirement
- SSR/on-demand rendering - rejected because GitHub Pages cannot host it

**Risks**:
- Dynamic content preview is not available without rebuild

**Mitigation**:
- Keep content local and low-frequency
- Use strict content schemas so builds fail early on invalid content

**Constitutional impact**: Positive. Maximizes GitHub Pages compatibility and simplicity.

## D04 - GitHub Pages Deployment Model

**Decision**: Use GitHub Actions as the only Pages source and deploy the static `dist/` output generated by Astro.

**Rationale**:
- Follows Astro and GitHub Pages guidance for static deployment
- Keeps the main repository as the single source of truth
- Allows full validation before deployment

**Alternatives considered**:
- Manual Pages publishing from a branch - rejected because it is less reproducible and easier to desynchronize
- Deployment via Higgsfield - rejected by the constitution

**Risks**:
- Incorrect `base` path can break assets on Project Pages

**Mitigation**:
- Centralize `siteUrl` and `basePath`
- Validate asset paths and direct deep links in E2E tests

**Constitutional impact**: Positive. Deployment remains repository-owned and reproducible.

## D05 - `site` and `base` Configuration Strategy

**Decision**: Centralize public URL configuration in one root config module consumed by `astro.config.mjs`, metadata generation, and robots generation.

**Rationale**:
- Supports both Project Pages and custom domain modes cleanly
- Avoids editing multiple files when the public URL changes
- Reduces path mistakes in canonicals, OG URLs, and sitemap references

**Alternatives considered**:
- Hardcode values directly in `astro.config.mjs` and page components - rejected because it multiplies edit points
- Derive everything from environment variables only - rejected as less maintainable for a small static site

**Risks**:
- Inconsistent use of config values across components

**Mitigation**:
- Use a single exported config object
- Prohibit hardcoded absolute URLs in page components

**Constitutional impact**: Positive. Improves Pages reliability and maintainability.

## D06 - Local Content Strategy

**Decision**: Use local, versioned content only: typed data modules for global company/site configuration and Astro content collections for services and projects.

**Rationale**:
- Matches the no-CMS requirement
- Makes content changes code-reviewed and auditable
- Supports build-time validation and publication gating

**Alternatives considered**:
- External headless CMS - rejected because it adds runtime/operational complexity and violates the first-version constraints
- Unstructured JSON blobs only - rejected because content collections offer stronger schema/query ergonomics for repeatable entries

**Risks**:
- Non-technical editors need repository access or mediated content updates

**Mitigation**:
- Keep content formats simple and documented in `quickstart.md`
- Prefer Markdown frontmatter for repeatable entries

**Constitutional impact**: Positive. Local content preserves repository ownership and runtime independence.

## D07 - Services and Projects Schema Strategy

**Decision**: Define strict content schemas in `src/content.config.ts` with explicit publication booleans and approval/authorization gates.

**Rationale**:
- Prevents invalid published entries from reaching production
- Encodes the constitution's truthfulness and visual honesty rules into the build
- Supports static path generation based only on eligible entries

**Alternatives considered**:
- Free-form Markdown with minimal validation - rejected because it risks broken or misleading pages
- Pure TypeScript arrays - rejected because content collections scale better for repeatable entries

**Risks**:
- Early schema strictness may slow initial content entry

**Mitigation**:
- Keep required fields minimal for draft entries
- Make public publication stricter than draft authoring

**Constitutional impact**: Strongly positive. This is the main technical guard against publishing misleading content.

## D08 - Image and Asset Strategy

**Decision**: Store publishable images locally, keep image sources under `src/assets/` whenever optimization is desired, and use Astro image components for responsive optimized output.

**Rationale**:
- `astro:assets` can infer dimensions and help reduce CLS
- Local assets preserve repository ownership and runtime independence
- Responsive formats help meet mobile performance budgets

**Alternatives considered**:
- Serve all images from `public/` - rejected because it bypasses optimization
- External image CDN - rejected because it adds runtime dependence and privacy/performance complexity

**Risks**:
- Large originals can still bloat the repository if unmanaged

**Mitigation**:
- Keep publishable derivatives in repo
- Preserve oversized originals outside the public asset path when needed
- Record provenance in a design asset log

**Constitutional impact**: Positive. Local, optimized, labeled assets preserve truthfulness and performance.

## D09 - SEO and Sitemap Strategy

**Decision**: Use `@astrojs/sitemap`, generated canonical metadata, semantic headings, `lang="pt-PT"`, a static 404 page, and conditional JSON-LD only when facts are confirmed.

**Rationale**:
- Official Astro sitemap integration fits static routes and dynamic routes generated by `getStaticPaths()`
- Centralized metadata reduces duplication and omission risk
- Conditional structured data avoids fabricating legal or business facts

**Alternatives considered**:
- Manual sitemap generation - rejected because it is harder to keep in sync
- Always-on LocalBusiness schema - rejected because required real company data is still pending

**Risks**:
- Missing `siteUrl` would weaken sitemap and canonicals

**Mitigation**:
- Treat `siteUrl` as a required deployment configuration value

**Constitutional impact**: Positive. Improves discoverability without manipulating SEO or inventing data.

## D10 - Unit Testing Scope

**Decision**: Use Vitest only for logic that materially benefits from unit coverage.

**Rationale**:
- Matches the proportional testing requirement
- Keeps the stack small
- Focuses unit tests on helpers such as URL construction, publication filters, and formatting utilities

**Alternatives considered**:
- No unit tests at all - rejected because some logic is easier and cheaper to verify outside E2E
- Blanket unit testing of static markup - rejected as artificial and low value

**Risks**:
- Contributors may overuse unit tests on presentation code

**Mitigation**:
- Restrict unit tests to pure logic in `src/utils/` and similar modules

**Constitutional impact**: Positive. Supports validation without overcomplicating the project.

## D11 - E2E and Accessibility Test Strategy

**Decision**: Use Playwright for critical flows and `@axe-core/playwright` for automated accessibility scanning inside those flows.

**Rationale**:
- Playwright covers Chromium, Firefox, and WebKit
- Works well with keyboard navigation and route validation
- Axe integration catches serious accessibility regressions early

**Alternatives considered**:
- Cypress - rejected because Playwright better matches the requested browser matrix and modern static-site CI usage
- Manual a11y review only - rejected because CI needs repeatable blocking checks

**Risks**:
- Browser tests can be slower or flaky if over-scoped

**Mitigation**:
- Limit E2E coverage to high-value user journeys
- Use mocked or controlled form-provider behavior where appropriate

**Constitutional impact**: Positive. Directly supports WCAG-oriented validation and pre-publication evidence.

## D12 - Performance Validation Strategy

**Decision**: Use Lighthouse CI with explicit score thresholds and budget monitoring on representative pages.

**Rationale**:
- Turns stated performance goals into enforceable CI assertions
- Allows repeated runs to reduce measurement variance
- Pairs well with a static site where performance regressions are usually bundle/asset driven

**Alternatives considered**:
- Manual Lighthouse runs only - rejected as non-repeatable
- No automated performance checks - rejected by the constitution's validation and performance principles

**Risks**:
- Lighthouse scores vary between runs

**Mitigation**:
- Use consistent CI settings and repeated runs
- Treat small fluctuations near the threshold as review signals, not automatic architecture changes

**Constitutional impact**: Positive. Provides measurable performance validation before publication.

## D13 - Link and Dependency Audit Strategy

**Decision**: Use automated link validation in CI and run `npm audit` as a baseline dependency security check.

**Rationale**:
- Broken internal links are a constitutional pre-publication failure
- `npm audit` gives low-cost security signal on the dependency set
- A GitHub Action-based link checker avoids adding another runtime package

**Alternatives considered**:
- Manual link clicking only - rejected as incomplete and not scalable
- Custom link-check scripts - rejected as unnecessary complexity

**Risks**:
- External links can flap intermittently

**Mitigation**:
- Separate critical internal-link failures from softer external-link handling if needed
- Prefer normal links instead of embedded third-party widgets

**Constitutional impact**: Positive. Supports reliability, safety, and validation gates.

## D14 - Contact Form Provider Comparison

**Decision**: Compare Formspree, Web3Forms, Static Forms, and `mailto:` before selecting a static-compatible provider.

### Comparison Matrix

| Option | GitHub Pages Compatible | No-JS Behavior | Secret Exposure | Spam Controls | Data / Retention Notes | Pros | Cons |
|--------|-------------------------|----------------|-----------------|---------------|------------------------|------|------|
| Formspree | Yes | Native HTML form POST | Public endpoint only; no frontend secret required | Built-in filtering, optional captcha/custom rules | Processes/stores submission data; privacy policy states processing in the US | Mature HTML form support, good progressive enhancement path, broad integrations | US data processing; vendor dashboard retention must be accepted |
| Web3Forms | Yes | Native HTML form POST | Public access key; explicitly documented as non-secret | Honeypot, hCaptcha, paid Turnstile/reCAPTCHA | Free plan includes 250 submissions/month, 30 days history; pricing page mentions retention policy up to 3 years | Simple setup, free tier, static-site friendly | Public key model may be less comfortable; retention/policy review still required |
| Static Forms | Yes | Native HTML form POST | Public API key model | CAPTCHA options incl. Turnstile/ALTCHA | Free plan forwards to email; optional storage on paid plans | Static-site compatible, privacy-friendly captcha options | Fewer details surfaced in fetched docs; more reliance on CAPTCHA configuration |
| `mailto:` | Technically yes | Depends on visitor mail client | No secret issue | None | No third-party storage | Zero vendor dependency | Poor UX, no server-side validation, unreliable on many devices, no delivery guarantees |

**Rationale**:
- All static providers remain operational without a custom backend
- `mailto:` is a fallback, not a primary form strategy

**Alternatives considered**:
- CRM-native embeds - rejected because they add complexity and often rely on proprietary scripts
- Self-hosted endpoint - rejected because GitHub Pages cannot run it

**Risks**:
- Any third-party form vendor introduces privacy and availability risk

**Mitigation**:
- Keep fields minimal
- Keep contact alternatives visible
- Centralize provider configuration so replacement is feasible

**Constitutional impact**: Neutral-to-positive. One external dependency is allowed if justified, minimal, and backed by fallback.

## D15 - Selected Contact Form Provider

**Decision**: Select Formspree for v1.

**Rationale**:
- Explicitly supports plain HTML forms with no JavaScript or server code
- Uses a public form endpoint instead of a frontend secret
- Provides server-side spam filtering and validation support
- Offers a straightforward path for progressive enhancement without forcing a JS SDK
- Matches the earlier clarification decision unless later legal review rejects US-based processing

**Alternatives considered**:
- Web3Forms - viable fallback, but public access-key framing and retention messaging are less aligned with a conservative first release
- Static Forms - viable fallback, but the currently fetched documentation is thinner for our evaluation set
- `mailto:` - kept only as resilience fallback

**Risks**:
- Submission data is processed by a US-based vendor
- Vendor outage can temporarily break the form

**Mitigation**:
- Document the vendor clearly in privacy content
- Keep the data set minimal
- Keep `mailto:` and `tel:` contact options permanently available
- Centralize provider endpoint configuration so a future swap stays low-cost

**Constitutional impact**: Acceptable. The external runtime dependency is minimal, justified, and fails gracefully without hiding contact information.

## D16 - Spam Protection and Fallback Strategy

**Decision**: Start with honeypot + provider-side spam filtering, no visual CAPTCHA by default, and visible fallback contacts on the same page.

**Rationale**:
- Aligns with accessibility and privacy goals
- Reduces friction for legitimate visitors
- Meets the requirement for basic anti-automation protection

**Alternatives considered**:
- CAPTCHA by default - rejected because it adds accessibility and privacy costs too early
- No spam control at all - rejected because public forms need basic abuse protection

**Risks**:
- Spam may still get through on a public endpoint

**Mitigation**:
- Reassess volume after launch
- Add a privacy-friendlier provider-side challenge later only if necessary

**Constitutional impact**: Positive. Keeps the user flow simple while preserving a baseline security posture.

## D17 - Higgsfield MCP Integration Policy

Higgsfield MAY be used later for moodboards, visual exploration, illustrative concepts, layout ideation, or code-reference inspiration, but it MUST NOT become part of the website's runtime, build pipeline, or deployment path.

### Mandatory Rules

- No Higgsfield repository, hosting, preview, iframe, embed, script, or runtime SDK may be required for the public website
- No Higgsfield credential or token may appear in repository files, CI, or frontend code
- Any selected Higgsfield output must be exported as a local file and committed to the main repository before use
- Any Higgsfield-derived asset must be tracked in `docs/design/asset-provenance.md`
- Generated or illustrative visuals must be labeled accurately and MUST NOT be presented as real LMDreams projects
- Higgsfield-generated code may only be used as reference; any adopted code must be adapted to Astro, reviewed, validated, and owned locally

### Practical Allowed Uses

- Moodboards and design direction exploration
- Illustrative placeholders clearly marked as such
- Reference compositions for hero sections or section ordering
- Non-runtime inspiration for tone, imagery, or layout

### Practical Forbidden Uses

- Hosting the production site or any required preview on Higgsfield
- Linking page assets directly from Higgsfield-hosted URLs
- Generating fictional portfolio evidence
- Using Higgsfield as a required CMS or asset CDN

**Constitutional impact**: Strongly positive. This policy operationalizes Principles I, II, III, and X.

## D18 - Asset Provenance and Licensing

**Decision**: Every publishable asset must have an origin record and licensing/authorization status.

**Rationale**:
- Prevents accidental publication of unlicensed or misleading visuals
- Makes later replacement of illustrative/generated assets straightforward
- Supports truthful communication and legal hygiene

**Alternatives considered**:
- Informal manual tracking outside the repo - rejected because it is hard to audit

**Risks**:
- Manual provenance logging can be skipped under time pressure

**Mitigation**:
- Make provenance logging a release gate in future tasks
- Keep file naming descriptive and origin enums explicit

**Constitutional impact**: Positive. Reinforces truthfulness, ownership, and validation principles.
